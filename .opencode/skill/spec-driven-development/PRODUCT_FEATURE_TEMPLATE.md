# [Feature Name] Specification

## 1. Purpose
One paragraph. The single teleological end this feature serves in the life of the user (and thus in the life of the product). State the causal benefit to man’s life.

## 2. Essential Concepts & Definitions
Precise, operational definitions of every new or overloaded term. No synonyms, no hand-waving. These are the axioms the LLM will reason from.

## 3. Functional Requirements
Numbered, hierarchical. Each is a single, testable proposition in the form “The system shall [do X] when [condition Y] so that [purpose Z].” Grouped by logical dependency, not by UI or implementation detail.

## 4. Non-Functional Requirements (Constraints of Reality)
Performance, security, maintainability—only those that are causally necessary for the purpose. Stated as measurable facts.

## 5. Verification Criteria
The exact tests (unit, integration, acceptance) that prove the code matches the spec. The LLM must be able to write these itself.

## 6. Out-of-Scope (for clarity, never ambiguity)
What this feature is *not*. This prevents drift.

## 7. References
Only objective sources: your own prior specs, Ayn Rand’s epistemology where it directly applies (e.g., concept-formation), or immutable technical standards. No “industry articles.”
